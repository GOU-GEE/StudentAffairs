# 智慧学工综合信息系统V2.0

## 一、 系统架构与工程规范

本系统采用极致优化的单体架构（Optimized Monolith），摒弃高运维成本的微服务，以代码级的工程严谨度和底层设计模式作为核心技术壁垒。

### 1. 核心技术栈

- **后端引擎**：Spring Boot 3 + Spring MVC + MyBatis-Plus。
- **数据持久层**：PostgreSQL 15（深度利用 JSONB 扩展能力）。
- **缓存与锁**：Redis 7.0（仅用于 Token 续期与轻量级分布式锁预案）。
- **前端渲染**：Vue 3 (Composition API) + Tailwind CSS + Element Plus（或同等现代组件库）。
- **权限中枢**：Sa-Token（轻量级 RBAC 拦截）。

### 2. 四大工程级高光点（答辩核心武器）

- **OOP 状态机引擎 (State Machine)**：所有复杂的业务审批（请假、奖助）严禁使用 `if-else`。必须抽象出全局状态枚举类与状态流转校验器，实现业务状态的受控转移。
- **AOP 动态数据脱敏与审计**：基于自定义注解 `@OperateLog` 和 `@DataMasking`，利用 Spring AOP 切面技术，实现无侵入式的操作日志落库与敏感字段（身份证号、手机号）的按权限脱敏渲染。
- **基于版本号的乐观锁防超发**：在处理勤工助学等高并发抢岗业务时，底层数据表引入 `version` 字段，利用 MyBatis-Plus 乐观锁机制解决并发冲突。
- **前端工程化与高阶组件**：前端摒弃面条代码，封装高复用的 `ProTable`（自动化分页搜索）和 `ProForm` 组件，并配合动态路由（Dynamic Routing）实现页面级的绝对权限隔离。

## 二、 角色与权限模型（RBAC）

系统实行严格的数据权限隔离策略，防范任何形式的越权访问。

1. **超级管理员 (Super Admin)**：系统最高权限，掌控全局配置与部门级管理员账号派发，拥有全量数据审计与监控权限。
2. **部门管理员 (Dept Admin)**：权限受限于所属学院/部门（`college_id`），负责本院系指标分配、全局学业数据清洗与最终业务流审批。
3. **教师 (Teacher)**：
   - **辅导员/班主任**：强绑定所属班级（`class_id`），数据权限仅限本班。负责日常初审、学业预警干预。
   - **跨部门教师（如就业指导）**：基于业务角色分配权限，不受班级限制。
4. **学生 (Student)**：仅限操作本人账号（`student_id`）及发起的关联业务。

## 三、 六大核心功能模块规格说明

### 1. 学生基本信息管理模块（动态画像与降维存储）

- **一站式画像大屏**：单页面瀑布流展示全量学生档案。包含姓名、学号、照片、联系方式、身份证、学院、专业、班级、宿舍、困难认定等级等。
- **核心技术实现**：
  - **非结构化数据降维**：针对“不少于2位的紧急联系人”、“非固定长度的病史、爱好及性格特征”，放弃传统的一对多子表关联，直接采用 **PostgreSQL 的 `JSONB` 类型**存储于 `student_profile` 主表，极大提升 I/O 效率。
- **关键信息变更审批**：联系方式、家庭住址的变动触发状态机流转，由辅导员进行初审拦截。
- **按权限流式导出**：引入 EasyExcel 配合权限校验器，辅导员与部门管理员只能导出自身数据权限内的字段。

### 2. 奖助服务管理模块（并发控制与严谨工作流）

- **业务闭环**：覆盖奖学金/助学金的发布、申请、初审、终审；勤工助学岗位的投递与录用。
- **核心技术实现**：
  - **防超发锁机制**：在勤工助学岗位报名环节，后端必须强制引入**乐观锁**校验，阻断高并发下的名额超卖现象。
  - **防刷单拦截**：系统自动校验学生的困难等级、纪律处分状态，不符合准入条件的在接口层直接抛出业务异常。

### 3. 学业发展支持模块（海量吞吐与自动化预警）

- **业务闭环**：成绩/评奖评优/违纪查询、中期鉴定、第二课堂成绩单上传审核。
- **核心技术实现**：
  - **纯异步海量导入**：抛弃繁重的消息队列。利用 Spring `@Async` 异步线程池配合 EasyExcel，实现十万级成绩单的非阻塞式解析入库，遇到脏数据直接抛弃并生成异常报告。
  - **定时预警引擎**：利用 `@Scheduled` Cron 任务，每日凌晨自动扫描 `scores` 表，将挂科数超标或连续绩点下滑的学生抽离至 `warning_log` 表，驱动辅导员的面谈邀约流程。

### 4. 职业规划模块（资源互斥与精准路由）

- **业务闭环**：内置生涯测评题库、就业指导预约、精准就业岗位推送。
- **核心技术实现**：
  - **排他性预约控制**：就业指导老师的时段预约接口，后端需利用数据库行级悲观锁 (`SELECT ... FOR UPDATE`) ，确保同一半小时的时段绝对不可被两人成功抢占。
  - **基于索引的岗位路由**：基于学生的 `major` 和 `career_goal` 字段，通过倒排或组合 SQL 查询，实现高效率的“千人千面”岗位大屏推荐。

### 5. 安全管理与预警模块（实时干预系统）

- **业务闭环**：外出报备、晚归登记、突发安全事件一键上报、安全知识测验。
- **核心技术实现**：
  - **SSE 实时红牌推送**：针对学生触发的“特急状况一键上报”或定时任务扫出的“超时未销假”，后端通过 Server-Sent Events (SSE) 建立单向长连接，向责任教师的 Web 端瞬时推送警报弹窗与高亮音效。

### 6. 沟通互动模块（极简通信通道）

- **业务闭环**：师生点对点私信、班/院/校三级公告发布（带已读未读追踪）、意见留言板、学术活动报名核销。
- **核心技术实现**：
  - **轻量级 WebSocket 通信**：摒弃复杂的群聊与多媒体文件同步。基于内存维护 `Session` 映射，仅支持纯文本的即时送达与离线落库，确保通信链路的绝对稳固和低延迟。

## 四、 接口与协同规范（API Contract）

为确保一月内极限交付，前后端双线并行开发必须遵守以下铁律：

1. **API 驱动**：所有模块开发前，必须在 Apifox 固化输入输出 JSON 格式。接口格式一经敲定，任何人修改必须通报对方。
2. **统一响应体**：后端必须返回标准的 `Result<T>` 泛型结构（含 `code`, `msg`, `data`）。前端基于统一状态码进行全局错误拦截。
3. **Mock 预案**：若后端接口延期，前端必须立刻切入本地 Mock 数据，确保演示流程不断档。
